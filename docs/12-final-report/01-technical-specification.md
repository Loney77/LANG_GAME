# Техническое задание

Документ составлен в соответствии со структурой ГОСТ 19.201-78 «Техническое задание.
Требования к содержанию и оформлению».

## 1. Введение

### 1.1. Наименование программы

«Сёз оюн» - мобильное приложение для обучения карачаевскому языку в игровой форме.

### 1.2. Краткая характеристика области применения

Программа предназначена для самостоятельного изучения и закрепления лексики
карачаевского языка. Область применения - образование, сохранение и популяризация
языков с сокращающейся сферой использования.

## 2. Основание для разработки

Разработка ведётся в рамках курсового проекта по дисциплине «Программная инженерия»
(СКФУ, направление 09.03.04), траектория В - мобильная разработка.

## 3. Назначение разработки

### 3.1. Функциональное назначение

Предоставление пользователю набора словесных игр на карачаевской лексике с
серверной проверкой ответов, подсчётом очков и рейтингом, а также средств
администрирования словаря.

### 3.2. Эксплуатационное назначение

Приложение эксплуатируется конечными пользователями на мобильных устройствах
под управлением Android и администратором - через защищённый программный интерфейс
сервера.

## 4. Требования к программе

### 4.1. Требования к функциональным характеристикам

Программа должна обеспечивать:

1. Регистрацию и аутентификацию пользователей с разграничением ролей (пользователь,
   администратор) на основе JWT.
2. Игру «Сёздл»: серверную проверку догадки и подсветку букв с учётом диграфов.
3. Игру «Анаграммы»: формирование набора букв и проверку собранного слова.
4. Игру «Викторина»: выбор правильного перевода из вариантов.
5. Игру «Кроссворд»: генерацию сетки пересекающихся слов и проверку заполнения.
6. Ежедневные задания.
7. Подсчёт очков и формирование рейтинга игроков с фильтрацией по виду игры и периоду.
8. Управление словарём и темами (CRUD) для администратора.
9. Кэширование словаря и заданий для работы при отсутствии сети.

### 4.2. Требования к надёжности

- Серверная проверка ответов исключает раскрытие правильных ответов клиенту.
- Некорректные запросы обрабатываются с возвратом стандартных кодов состояния HTTP
  (400, 401, 403, 404).
- Целостность данных обеспечивается ограничениями базы данных в третьей нормальной форме.

### 4.3. Требования к составу и параметрам технических средств

| Компонент | Требование |
|-----------|-----------|
| Клиент | Мобильное устройство Android 8.0+ |
| Сервер | JRE 17+, 512 МБ ОЗУ и выше |
| База данных | PostgreSQL 16 |

### 4.4. Требования к информационной и программной совместимости

- Сервер: Java 17, Spring Boot 3, PostgreSQL, контракт REST/OpenAPI.
- Клиент: React Native (Expo), взаимодействие с сервером по HTTPS/JSON.
- Аутентификация: JWT в заголовке `Authorization`.

### 4.5. Требования к безопасности

- Хранение паролей в виде криптографических хэшей.
- Подписанные JWT с ограниченным сроком действия.
- Разграничение доступа к административным операциям по роли.

## 5. Требования к программной документации

В состав документации входят: техническое задание, документация по этапам
проектирования (разделы `docs/00`–`docs/10`), руководство пользователя, руководство
администратора, пояснительная записка.

## 6. Технико-экономические показатели

Оценка трудоёмкости и сроков приведена в документах [WBS](02-wbs.md),
[календарный план](03-gantt-chart.md) и [оценка COCOMO](04-cocomo-estimation.md).

## 7. Стадии и этапы разработки

Разработка разбита на восемь этапов в соответствии с календарным планом
(см. [03-gantt-chart.md](03-gantt-chart.md)): инициация, требования, архитектура,
проектирование базы данных, детальное проектирование, реализация ядра, рефакторинг,
реализация интерфейса и завершение.

## 8. Порядок контроля и приёмки

- Серверный код покрывается модульными тестами (целевое покрытие - свыше 40% по JaCoCo).
- Эндпоинты проверяются через Swagger UI, включая контроль кодов состояния и
  разграничения доступа.
- Игровые сценарии проверяются на клиенте до экрана результата, включая работу
  при отсутствии сети.
