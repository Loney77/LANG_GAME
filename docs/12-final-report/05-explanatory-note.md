# Пояснительная записка

## 1. Введение

Настоящая записка описывает результаты разработки мобильного приложения «Сёз оюн»
для обучения карачаевскому языку в игровой форме. Проект выполнен в рамках курсовой
работы по дисциплине «Программная инженерия» по траектории мобильной разработки
(React Native и Java Spring Boot).

## 2. Постановка задачи

Карачаевский язык относится к языкам с сокращающейся сферой использования.
Доступных интерактивных цифровых инструментов для его изучения практически нет.
Требовалось разработать клиент-серверное приложение, которое в игровой форме
расширяет словарный запас пользователя и поддерживает мотивацию через систему
очков и рейтинг, а также предоставляет администратору средства управления словарём.

## 3. Функциональные возможности

- Регистрация и аутентификация пользователей с разграничением ролей по JWT.
- Четыре словесные игры: «Сёздл», «Анаграммы», «Викторина», «Кроссворд».
- Ежедневные задания, подсчёт очков и рейтинг игроков.
- Управление словарём и темами для администратора (CRUD).
- Кэширование словаря и заданий для работы без подключения к сети.

## 4. Архитектура

Система построена по архитектурному шаблону **PCMEF** с направленностью
зависимостей строго сверху вниз (Presentation → Control → Mediator → Entity →
Foundation).

| Слой | Реализация |
|------|-----------|
| Presentation | Мобильный клиент React Native (TypeScript) и контракт REST/OpenAPI |
| Control | Контроллеры Spring (`AuthController`, `WordController`, `GameController` и др.) |
| Mediator | Сервисы бизнес-логики (`SozdlService`, `AnagramService`, `QuizService`, `CrosswordService`, `ScoringService`, `LeaderboardService`) |
| Entity | JPA-сущности с доменными методами и токенизатор алфавита |
| Foundation | Репозитории Spring Data JPA и мапперы Data Mapper |

Подробное описание приведено в разделах [02-architecture](../02-architecture/) и
[04-detailed-design](../04-detailed-design/).

## 5. Модель данных

Доменная модель включает сущности `User`, `Theme`, `Word`, `GameType`, `Puzzle`,
`QuizQuestion`, `GameSession`. База данных спроектирована в третьей нормальной форме;
схема и начальные данные применяются миграциями Flyway. ER-диаграмма и DDL приведены
в разделе [03-database](../03-database/).

## 6. Ключевые технические решения

### 6.1. Токенизатор карачаевского алфавита

Диграфы карачаевского алфавита (гъ, къ, нъ, нг, дж) считаются одной буквой.
Токенизатор разбивает слово на буквы с учётом диграфов; он используется при подсчёте
длины слова, подсветке букв в «Сёздл» и формировании наборов для анаграмм.
Решение зафиксировано в [ADR-003](../02-architecture/adr/ADR-003-alphabet-tokenizer.md).

### 6.2. Серверная проверка ответов

Правильные ответы и логика проверки находятся только на сервере. Клиент получает
лишь результат проверки (например, подсветку букв), что исключает раскрытие ответа
и попытки обхода правил.

### 6.3. Аутентификация без состояния

Применяется аутентификация на основе подписанных JWT, передаваемых в заголовке
запроса. Решение описано в [ADR-004](../02-architecture/adr/ADR-004-jwt-stateless-auth.md).

## 7. Применённые паттерны

| Паттерн | Применение |
|---------|-----------|
| Data Mapper | Отделение доменных сущностей от DTO и слоя персистентности (мапперы в Foundation) |
| Identity Map | Контекст персистентности Hibernate как кэш первого уровня в рамках транзакции |
| Lazy Load | Отложенная загрузка связей JPA |

Подробности — в разделе [07-refactoring](../07-refactoring/).

## 8. Программный интерфейс

Сервер предоставляет более восьми REST-эндпоинтов, документированных по стандарту
OpenAPI и доступных через Swagger UI. Описание интерфейса приведено в разделе
[09-api](../09-api/).

## 9. Тестирование

Ключевая бизнес-логика покрыта модульными тестами (JUnit 5, Mockito): токенизатор
алфавита, проверка догадки «Сёздл», подсчёт очков. Целевое покрытие серверного
кода — свыше 40% по отчёту JaCoCo. Стратегия и состав тестов описаны в разделе
[06-testing](../06-testing/).

## 10. Технико-экономические показатели

Декомпозиция работ приведена в [WBS](02-wbs.md), сроки — в
[календарном плане](03-gantt-chart.md), оценка трудоёмкости — в
[расчёте COCOMO](04-cocomo-estimation.md). Фактический объём оригинального кода
составил около 4.3 тысяч строк.

## 11. Заключение

Разработано клиент-серверное мобильное приложение, реализующее поставленные цели:
интерактивное изучение карачаевской лексики через словесные игры, серверную проверку
ответов, систему очков и рейтинг, а также администрирование словаря. Соблюдены
требования архитектуры PCMEF, нормализации базы данных, документирования API и
покрытия кода тестами. Дальнейшее развитие возможно в направлениях расширения
словаря, обучающего слоя (интервальное повторение, достижения) и публикации
приложения в магазине приложений.
