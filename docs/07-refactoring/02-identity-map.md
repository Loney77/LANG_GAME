# Паттерн Identity Map

**Назначение (методичка):** обеспечение уникальности объектов в сессии — один
экземпляр объекта на запись БД в пределах одной сессии/транзакции.

## Реализация: контекст персистентности Hibernate (L1-кэш)

В проекте Identity Map обеспечивается **контекстом персистентности JPA/Hibernate**
(first-level cache). В пределах одной транзакции повторная загрузка одной и той же
строки возвращает **тот же** экземпляр объекта:

```java
@Transactional
public void demo(Long id) {
    Word a = wordRepository.findById(id).orElseThrow();
    Word b = wordRepository.findById(id).orElseThrow();
    assert a == b;          // один и тот же объект — Identity Map
}
```

Это гарантирует:
- отсутствие рассинхронизации состояния (нет двух разных копий одной сущности);
- меньше обращений к БД (повторные `find` берутся из кэша сессии);
- корректную работу `Unit of Work` Hibernate при flush.

## Где это работает в коде

Транзакционные границы заданы в слое Mediator аннотацией `@Transactional`
(например, `SozdlService.evaluateGuess`): загруженные в рамках вызова `Puzzle`,
`Word`, `GameSession` уникальны в пределах этой транзакции.

## Почему не своя реализация

Своя `Map<Id, Entity>` дублировала бы то, что Hibernate уже делает корректно
(с учётом flush/dirty-checking). Поэтому используем штатный механизм и опираемся
на него осознанно, задавая правильные транзакционные границы.

> Связано с паттерном [Lazy Load](03-lazy-load.md) — оба обеспечиваются
> контекстом персистентности.
