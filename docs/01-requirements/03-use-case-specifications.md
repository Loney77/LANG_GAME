## UC4. Отправить догадку в Сёздл

| Поле | Значение |
|------|----------|
| **Актор** | Игрок (роль `USER`) |
| **Цель** | Проверить догадку игрока и получить обратную связь по буквам |
| **Предусловия** | Игрок аутентифицирован (валидный JWT); существует активное daily-задание Сёздл |
| **Постусловия** | Попытка учтена; при победе/исчерпании попыток - создана/обновлена сессия игры |
| **Триггер** | Игрок вводит слово из 5 букв и нажимает «Проверить» |

### Основной поток

1. Игрок вводит слово-догадку (5 букв карачаевского алфавита).
2. Клиент отправляет `POST /api/games/sozdl/guess` с `{ puzzleId, guess }` и JWT.
3. Сервер (Control) валидирует формат запроса.
4. Сервер (Mediator, `SozdlService`):
   1. токенизирует догадку по алфавиту (диграфы = 1 буква);
   2. проверяет, что догадка есть в словаре (Foundation);
   3. сравнивает с целевым словом задания, вычисляет статус каждой плитки
      (🟩 на месте / 🟨 есть в слове / ⬜ отсутствует);
   4. увеличивает счётчик попыток.
5. При совпадении или исчерпании лимита попыток - фиксирует `GameSession`
   (статус `WIN`/`LOSS`, очки через `calculateScore()`).
6. Сервер возвращает массив статусов плиток, номер попытки и флаг завершения.
7. Клиент отображает подсветку букв.

### Альтернативные потоки

- **A1. Слова нет в словаре.** Шаг 4.2: сервер возвращает `422` «слово не найдено»,
  попытка не засчитывается.
- **A2. Неверная длина/символы.** Шаг 3: `400 Bad Request`.
- **A3. Лимит попыток исчерпан ранее.** Сервер возвращает `409 Conflict`
  «задание уже завершено».
- **A4. Нет/просрочен токен.** `401 Unauthorized`.

## UC11. Управление словарём (CRUD)

| Поле | Значение |
|------|----------|
| **Актор** | Администратор (роль `ADMIN`) |
| **Цель** | Создавать, изменять, удалять и искать словарные статьи |
| **Предусловия** | Аутентифицирован как `ADMIN` |
| **Постусловия** | Словарь изменён; `letterCount` пересчитан сервером |
| **Триггер** | Действие администратора в интерфейсе управления |

### Основной поток (создание)

1. Администратор заполняет форму: слово, перевод, тема.
2. Клиент отправляет `POST /api/words` с JWT администратора.
3. Сервер (Control) проверяет роль `ADMIN` и валидирует данные.
4. Сервер (Mediator) вычисляет `letterCount` токенизатором, сохраняет (Foundation).
5. Возвращается `201 Created` с созданной статьёй.

### Альтернативные потоки

- **A1. Дубликат слова.** Шаг 4: `409 Conflict`.
- **A2. Недостаточно прав (роль USER).** Шаг 3: `403 Forbidden`.
- **A3. Удаление слова, используемого в заданиях.** Запрет или мягкое удаление
  (решается на Этапе 3 ограничениями FK).

## UC2. Вход (JWT) - кратко

| Поле | Значение |
|------|----------|
| **Актор** | Игрок / Администратор |
| **Основной поток** | `POST /api/auth/login` → сервер проверяет email+пароль (BCrypt) → возвращает JWT с ролью |
| **Альтернатива** | Неверные учётные данные → `401 Unauthorized` |
